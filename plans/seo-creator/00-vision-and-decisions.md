# SEO Creator ‚Äî Vision & Top-Level Decisions

The SEO Creator is an agent-driven engine that produces and governs **multi-page SEO/GEO content hubs** ‚Äî a pillar page plus a funnel-staged cluster of long-form, E-E-A-T-grade guides ‚Äî for agency clients whose topics are often YMYL (memory care, finance, health). It serves the operator (an editor who shapes briefs and releases pieces) and the client (who reviews and signs off on the rendered hub), with a credentialed reviewer holding the non-overridable release authority on regulated content. It wins not because the LLM writes faster, but because the **harness is the moat**: a non-compensatory publish gate, a cross-model faithfulness check, a fail-closed lifecycle FSM, and per-tenant voice/corpus isolation that no "fast printer" competitor ships. The deliverable is a durable, crawlable, schema-marked asset that compounds ‚Äî built once, served to humans, search crawlers, AI answer engines, and (later) the product's own RAG.

The strategic insight that decides the build: the content engine **already exists on `origin/preview`** (PRs #1668-1684 ‚Äî the 22 scorers, the cross-model faithfulness gate, the non-compensatory `seo-gate`, the fail-closed `lifecycle-fsm`, and the `content_clients`/`content_pieces`/`content_piece_versions`/`voice_specs` Drizzle schema). We **port** that deterministic core verbatim as host-side platform primitives. *(‚öÝÔ∏è The original "re-author the four producer prompts as AI SDK v6 system prompts" plan is **SUPERSEDED by DECISIONS.md D5/D9**: because the locked runtime is the real Claude Agent SDK harness, the four `seo-copywriter` producers ‚Äî strategist, assistant, writer, audit ‚Äî run **directly as their literal `SKILL.md` skills**, kernel-backed via the host-side `/content/api/{brief,draft,audit,publish}` routes, not re-authored as in-process prompts.)* That collapses the headline cost from "rebuild an engine" to "wire the existing suite + the kernel route contract into the worker."

## The Five Questions, Answered

### 1. What the interface looks like

**A three-zone agent canvas modeled on videogen's `StudioCanvas`, minus the operator-only controls.** LEFT is an Agent chat panel streaming real token deltas ‚Äî live thinking and tool-use ("fetching SERP, running faithfulness gate, Stage-A clean, scoring 8 dimensions") ‚Äî so it visibly behaves like the Claude harness. CENTER is the artifact: a side-panel markdown editor (the artifacts pattern) for the content piece, toggling to a live SSR preview of the rendered page. RIGHT is an Inspector showing the two-stage gate scorecard (Stage-A vetoes as red blocking chips with stable codes, then Stage-B 8-dim 0-100 bars with the verdict band, faithfulness visibly dominant) plus version history. Generation is **brief-first**: the human reviews and refines the typed brief (intent observed from a live SERP fetch, cluster placement, entities, `isYmyl`/`reviewerRequired` flags) ‚Äî not the 2,200-word draft, because getting the brief right makes the draft bookkeeping. Visuals match the existing `apps/agents` convention: Tailwind v4 `foreground`/`background` tokens, the emerald/blue/amber/red verdict palette, `font-mono` eyebrow + `font-display` headings, inline `Link`-pill tab-nav.

### 2. What the output artifact looks like

**The unit is a `content_piece` ‚Äî a long-form E-E-A-T markdown article carrying a cluster role** (`pillar` | `cornerstone` | `spoke`/guide | `FAQ` | `checklist`) with `meta_description`, `excerpt`, `faq_data` (FAQPage JSON-LD), an 8-dimension scorecard, `brief_snapshot`, `version`, `status`, `is_ymyl`, and a credentialed `author_id`. The **deliverable is a cluster of ~8 such pieces forming a hub**: one pillar ("Memory Care in Skagit County, WA") plus funnel-staged spokes (Awareness ‚Üí Consideration ‚Üí Decision) realized as internal links and a curated resource-library homepage. Each piece carries intrinsic **E-E-A-T** (named accountable byline, source-grounding to named authorities like the Alzheimer's Association/NIA, medical disclaimer, trust/license badge), **GEO/AI-readiness** (FAQ schema, question-style headings, answer-first capsules, quotable fact sentences), and **SEO** (topic clustering, pillar/cluster internal linking, local geo-targeting, funnel-stage keywords). It is **content, not runnable code**. *(‚öÝÔ∏è The original gloss ‚Äî "which is precisely why Sandbox/Agent-SDK code-execution fidelity is unnecessary" ‚Äî is **SUPERSEDED by DECISIONS.md D5/D9**. The locked runtime *does* self-host the Claude Agent SDK on a Vercel Sandbox, but it uses the Sandbox to host the **autonomous loop process and the real `SKILL.md` harness**, not to execute the content as code; the artifact remains markdown + JSON-LD, and the gate stays host-enforced behind the `/content/api/*` routes.)*

### 3. Whether it resembles anything we've built

**Yes ‚Äî strongly, on two fronts, and explicitly NOT the removed retirement-pilot brochure site.** (1) The **engine** is the `apps/agents` content/`seo-copywriter` ContentEngine: the 22 deterministic scorers (flesch-kincaid, keyword-density, passive-voice, content-score, meta/og/faq generators), the cross-model faithfulness gate, the voice gate, and ‚Äî on `origin/preview` ‚Äî the `seo-gate`, `lifecycle-fsm`, and `content_pieces`/`voice_specs` schema. We port these, not reinvent. (2) The **UI** borrows the videogen agent canvas (`PinOverlay`, `PreviewClickHandler`, `useIframePinDrop`, `ChatEdit`, `VersionHub`, `ApprovalBeat` ‚Äî all verified present locally) for streaming generation, conversational fine-tune, versioning, and client on-screen feedback. The **output** resembles the live Whispering Willows demo (a multi-page content hub) ‚Äî a different deliverable from the single-tenant retirement-pilot brochure, which had no gate, no cluster model, and no YMYL governance.

### 4. How it looks presented to clients

**As a tokenized, workspace-scoped hosted live preview of the actual SSR-rendered hub.** The client judges the pillar page and guides exactly as Google, readers, and AI answer engines will see them ‚Äî full body in initial HTML, FAQ JSON-LD, the curated resource-library homepage (hero, statistic, named three-stage cluster, guide-card grid, tour CTA + license badge) ‚Äî paired with a SERP-snippet preview so they also see the search-result appearance. The client surface is **deliberately separate** from the operator surface: no credits, no Improve-Draft, no raw markdown export ‚Äî a clean rendered page, a version history, and (for a premium client) a curated "Reviewed & graded" trust strip. A PDF/print leave-behind (e.g. the printable 12-Question Tour Checklist) is a **secondary export, not the review surface** ‚Äî the client must judge the page as crawlers do, which a PDF cannot show.

### 5. How clients give on-screen feedback

**On the hosted preview itself, at two granularities.** (a) Free-form **pinned comments** anchored to actual page elements (reusing `PreviewClickHandler` click-to-pin + `PinOverlay` markers + `useIframePinDrop` iframe anchoring, with a Vercel-toolbar-style thread model: pinned location + messages + open/resolved status); and (b) **section-level Approve / Request-changes** verbs on each H2 (reusing `ApprovalBeat`'s pill + primary/secondary buttons). A "Request changes" comment **routes straight into the agent's conversational edit loop** ‚Äî the comment becomes the instruction, the agent emits a bounded body diff, applies it as a new auto-version, re-runs the full gate, and streams a one-line "what changed" summary, so feedback becomes an auditable, reversible version rather than an email. Approval-cycle time and open-thread count ("approval debt") are instrumented per client, because RELEASE ‚Äî not generation ‚Äî is the binding constraint. **Client approval is advisory on hard gates**: a client can never approve past a YMYL/faithfulness/thin-content veto, which stays non-overridable canon held by the credentialed reviewer.

## What "on-thesis" means

The product is on-thesis only if it treats the **LLM as the replaceable middle and the harness as the moat**. The "gate-is-the-wedge" principle is load-bearing: a draft scoring 95 on SEO, voice, and layout but failing any Stage-A floor is **non-publish**, because the floors are boolean vetoes evaluated *outside* the weighted sum, not heavy weights inside it. The UI must therefore show **"eligible" and "published" as distinct states** ‚Äî a high composite routes a piece to the human release gate; it does not open the door to the public web.

For YMYL clients like Whispering Willows (squarely YMYL), governance turns to eleven: `is_ymyl` is persisted at brief time from auditable signals and read authoritatively from the row at gate/FSM time (the judge may add the flag, never remove it); a named, credentialed **Person** byline (resolved server-side from `author_id`, closing the inherited `request.author` trust hole), citations to named authorities, and a recorded human release are all required before any piece reaches `status='published'`. A skipped, thrown, or timed-out eval **blocks** ‚Äî there is no autopilot. The engine must be architecturally incapable of synthesizing reviews or testimonials, and a backup-reviewer path must exist before any regulated client goes live, because the credentialed reviewer ‚Äî not cost ‚Äî is the binding constraint on YMYL publish rate.

Two disciplines keep this from quietly rotting, and they are non-negotiable: **(1) enforce every Stage-A veto and `canPublish()` in host code outside the agent loop** ‚Äî the agent gets read-only scoring tools and can never reason past a veto; and **(2) capture the live Whispering Willows hub as a human-labeled golden set BEFORE writing a single prompt**, and regress every prompt/model bump against it ‚Äî methodology-fidelity regression is the one failure mode that will not show up in CI, and catching it is what decides whether this build succeeds or silently degrades.
