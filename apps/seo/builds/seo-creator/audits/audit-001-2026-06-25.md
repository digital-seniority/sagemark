## Critical

### A.005.1 — `content_clients` has no RLS (anon tenancy-map leak)
- **Location:** `packages/schema-flywheel/src/content.ts` + migrations `0030/0031/0032` — no `ALTER TABLE content_clients ENABLE ROW LEVEL SECURITY` anywhere.
- **Finding:** With Postgres' RLS-disabled default and Supabase's standard anon grant, the anon role can `SELECT` every tenant's `content_clients` row (client name, `blog_slug`, `workspace_id`) — leaking the exact workspace↔client tenancy map the whole P0.S.1 boundary exists to protect. The RLS contract test never checks this table.
- **Recommendation (A.005.1 PR):** `ALTER TABLE content_clients ENABLE ROW LEVEL SECURITY` + a workspace-scoped policy (mirroring the published-only pattern, but `content_clients` likely has NO anon-readable rows at all → fail-closed no-policy), AND add `content_clients` to the zero-anon-row assertion list in `rls-contract.test.ts`. **Never auto-merges (A-finding + tenant-isolation path).**

## High (→ active risks; A.005.x PRs where noted)

### A.005.2 — Governance spec contradicts the locked decision + shipped code
- **Location:** `plans/seo-creator/01-architecture.md:9-30`, `plans/seo-creator/00-vision-and-decisions.md:13` vs `plans/seo-creator/DECISIONS.md` D5/D9.
- **Finding:** The architecture/vision specs still assert "Approach B — native AI SDK in-process; never self-host the Agent SDK and never use a Sandbox for content," but DECISIONS.md D5/D9 override to the opposite (self-hosted Claude Agent SDK worker on Vercel Sandbox), and the merged code + the P0.W.1 spike follow the override. The two governing docs were never reconciled — the doc a future implementer reads first directly contradicts the locked runtime, the highest-risk drift because it misdirects the still-unbuilt worker-host (P0.W.2) and SSE-transport lanes.
- **Recommendation (A.005.2 PR, docs):** stamp `01-architecture.md` §1-3 + `00-vision-and-decisions.md` row 1 as **superseded by D5/D9**; rewrite the topology to the worker→`apps/seo`→browser split the code implements. **Critical-adjacent for P0.W.2 correctness — resolve before P0.W.2 spec is handed to an agent.**

### A.005.3 — Faithfulness + voice gates bypass the metered Gateway (OpenRouter direct)
- **Location:** `packages/core/src/gates/faithfulness-gate.ts:90,257,267` + `voice-gate.ts:71,195,237` (`OPENROUTER_ENDPOINT`, `OPENROUTER_API_KEY`, raw fetch) vs `engineering-rfc.md §2` ("every model call accounted; worker never holds a raw provider key") + DECISIONS D4 (cost ledger from live Gateway usage).
- **Finding:** PR 002 ported the two LLM gates verbatim, but they hit OpenRouter directly instead of routing through the `resolveGatewayModel()` seam PR 001 built for metering. They run host-side (via `/content/api/audit`), so this is host-side un-metered egress (not a Sandbox escape) — but every faithfulness/voice call is invisible to the D4 ledger, and the drafter≠verifier invariant, while preserved (haiku verifier), is enforced via a hardcoded literal (`GATE_MODEL`) rather than the canonical `config/models.ts` id.
- **Recommendation (A.005.3 PR + DR):** re-point both gates through `resolveGatewayModel(VERIFIER_MODEL_ID, …)`; remove the OpenRouter endpoint/key; import the canonical verifier id. Record a DR for the interim posture if migration is deferred (cross-ref RFC §2 + D4; the D4 ledger PR 020 will otherwise be built on a false "all spend flows through Gateway" assumption).

### A.005.4 — No CI executes the tests (and the RLS suite is invoked by nothing)
- **Location:** root `package.json` (no `test` script), `turbo.json:30` (`test` task never invoked), no `.github/` workflow; `apps/seo/test/tenancy/rls-contract.test.ts:43` uses `node:test` and is excluded from `apps/seo/vitest.config.ts:16`.
- **Finding:** Nothing guards against a regression landing — there is no CI, and even locally the RLS/tenancy contract suite (the only coverage for the published-only anon policy, the release/signoff split, FK ON DELETE RESTRICT) runs under a runner no script invokes. STATE learning #5 already self-flags the related "worker-env lint fails the build" AC#3 gap; this is the broader root cause.
- **Recommendation (A.005.4 PR, CI):** add root `"test": "turbo run test"` + a GitHub Actions workflow running `pnpm test` on PRs to `preview`/`main`; wire `node --test test/tenancy/` into the `apps/seo` test script so Tier-1 RLS runs unconditionally; invoke `worker-env-lint` in the build (closes AC#3).

### A.005.5 — Behavioral RLS assertions never execute (Tier-2 skips without Postgres)
- **Location:** `apps/seo/test/tenancy/rls-contract.test.ts:311-314` (Tier-2 gated on `DATABASE_URL`/`RLS_TEST_PG_CONTAINER`).
- **Finding:** All 8 Tier-2 behavioral RLS assertions (anon row-visibility, cross-tenant isolation, CHECK/FK enforcement) skip rather than run with no PG wired — an open Tier-3 NEEDS-INPUT from `checkpoints/run-002` never followed up (distinct from the P0.W.1 live-Sandbox Tier-3, which IS resolved). The RLS guarantees are currently unproven by execution.
- **Recommendation:** active risk → provision a Supabase branch / ephemeral Docker pg in CI and set `DATABASE_URL`; fold into **P0.S.2** (and A.005.4's CI work). Combined with A.005.1, the tenancy boundary is the build's biggest unproven surface.
